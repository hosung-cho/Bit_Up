## 프로젝트 개요

# 🚀 프로젝트: 나만의 RISC-V MCU (SERV SoC) 제작

### 1. 시스템 개요 (System Overview)

- **두뇌 (CPU):** **SERV 코어** (RV32I 명령어 셋을 지원하는 비트 직렬 구조의 초소형 코어)
- **타겟 공정:** ETRI 0.5um CMOS (제한된 면적과 **28핀 SOP 패키지**라는 극한의 물리적 제약)
- **핵심 해결 과제:** 칩 내부 면적 및 핀 수 부족으로 인해 메모리를 내장할 수 없으므로, **라즈베리파이 피코(RP2040)를 'SPI 기반 외부 메모리 에뮬레이터'로 활용**하여 극복.

### 2. 하드웨어 스펙 및 주변장치 (Peripherals)

단순한 연산 장치를 넘어, 외부 세계와 소통할 수 있는 완벽한 SoC(System on Chip) 구성을 갖춥니다.

- **통신 및 디버깅:** **UART** (PC 터미널 연결용, 1순위 구현 목표)
- **범용 제어:** **GPIO** (LED 제어, 버튼 입력 등)
- **시간 제어:** **Timer** (정확한 지연 생성) & **PWM** (모터 제어, 부저 음향)
- **외부 확장:** **사용자용 SPI** (디스플레이 등) & **I2C** (온습도, 가속도 등 다중 센서 통신)
- **시스템 제어:** **WDT** (웟치독 타이머, 시스템 다운 시 자동 리셋) & **인터럽트 컨트롤러**

### 3. 통합 메모리 맵 (Memory Map)

SERV 코어가 `Wishbone` 버스를 통해 접근할 4GB의 공간을 가장 효율적인 하드웨어 디코딩 구조로 나누었습니다.

| **주소 영역** | **대상 모듈** | **특징** |
| --- | --- | --- |
| **`0x0000_0000 ~ 0x00FF_FFFF`** | **External Memory (Pi Pico)** | **16MB 할당.** 상위 8비트(`0x00`)만 검사하면 되어 하드웨어 게이트 소비가 가장 적음. |
| **`0x4000_0000`** 대역 | **UART** | 송수신 레지스터 (디버깅) |
| **`0x4000_1000`** 대역 | **GPIO** | 입출력 방향 및 데이터 제어 |
| **`0x4000_2000 ~ 7000`** 대역 | **Timer, PWM, SPI, I2C, WDT, Int. Ctrl** | 각 주변장치에 4KB씩 넉넉하고 규칙적으로 할당. |

### 4. 개발 및 검증 파이프라인 (Verification Flow)

실제 실리콘 칩으로 굽기 전에 "반드시 100% 동작한다"는 확신을 얻기 위한 검증 전략입니다.

1. **소프트웨어 코-시뮬레이션 (Verilator):** 베릴로그(Verilog) 설계를 초고속 C++ 코드로 변환하여 PC 환경에서 피코 동작을 가상으로 시뮬레이션 및 디버깅.
2. **하드웨어 코-에뮬레이션 (FPGA + Pico):** 설계한 SoC를 FPGA(Cyclone IV 등)에 올리고, 실제 피코와 점퍼 선으로 연결해 실시간 속도로 동작 확인.
3. **오픈소스 툴체인 합성 (Yosys ~ Magic):** 검증이 끝난 베릴로그 코드를 ETRI 0.5um 표준 셀에 맞춰 합성(Synthesis)하고 자동 배치/배선(P&R)을 거쳐 최종 GDS 도면 추출.

## 하드웨어 스펙 및 주변장치 (Peripherals) 정의

### 🧠 코어 및 기본 환경 (Core & Environment)

- **CPU 코어:** **SERV Core** (RV32I 명령어 셋 지원, 극소면적 비트 직렬 구조)
- **패키지 타겟:** **28핀 SOP** (ETRI 0.5um 공정, 전원 핀 제외 시 가용 GPIO 약 20여 개)
- **메모리 전략:** 내부 메모리 대신 **라즈베리파이 피코(RP2040)를 외부 메모리로 사용**하여 공간 및 핀 수 한계 극복.

### 🔌 주변장치(Peripherals) 최종 구성 리스트

**1. 외부 메모리 인터페이스 (SPI Bridge)**

- **역할:** 코어(Wishbone 버스)와 외부 피코 메모리 사이를 연결하는 전용 고속 직렬 통신.
- **사용 핀:** 4핀 (SCK, MOSI, MISO, CS_MEM)

**2. UART (범용 비동기 송수신기)**

- **역할:** PC와 연결하여 터미널 창에 `printf` 출력을 띄우거나 키보드 입력을 받는 가장 확실한 디버깅 도구.
- **사용 핀:** 2핀 (TX, RX)

**3. GPIO (범용 입출력 포트)**

- **역할:** MCU의 기본적인 '손발' 역할. LED를 켜고 끄거나 외부 버튼 입력을 읽어옴.
- **사용 핀:** 남는 핀 전체 할당 (약 10~12핀 내외 예상)

**4. Timer (타이머)**

- **역할:** `delay_ms()` 같은 정확한 시간 지연을 만들거나, 주기적으로 특정 작업을 실행하기 위한 내부 카운터.
- **사용 핀:** 없음 (칩 내부 논리 회로)

**5. PWM (펄스 폭 변조)**

- **역할:** 서보 모터의 각도/속도를 제어하거나, 부저(Buzzer)를 울리고 LED 밝기를 부드럽게 조절하는 아날로그적 제어 장치.
- **사용 핀:** 최소 2핀 (2채널 구성 시)

**6. User SPI (사용자 범용 SPI)**

- **역할:** 메모리용 SPI와 별개로, 사용자가 외부 OLED 디스플레이나 고속 센서를 연결할 때 사용하는 통신 포트.
- **사용 핀:** 4핀 (단, SCK/MOSI/MISO는 메모리용과 공유하고 CS 핀만 분리하면 핀 절약 가능)

**7. I2C (I2C 통신 제어기)**

- **역할:** 단 2개의 선만으로 온습도 센서, 가속도 센서, EEPROM 등 다양한 모듈을 동시에 주렁주렁 연결할 수 있는 강력한 센서 통신망.
- **사용 핀:** 2핀 (SDA, SCL)

**8. WDT (Watchdog Timer, 웟치독 타이머)**

- **역할:** 소프트웨어 버그로 무한 루프에 빠지거나 외부 노이즈로 칩이 멈췄을 때, 스스로 판단하여 칩을 강제 재부팅시키는 안전장치.
- **사용 핀:** 없음 (칩 내부 논리 회로)

**9. Interrupt Controller (인터럽트 컨트롤러)**

- **역할:** 주변장치(UART 데이터 수신, 타이머 완료 등)에서 발생하는 긴급 이벤트를 모아서 CPU에 알려주는 SoC의 핵심 지휘소.
- **사용 핀:** 없음 (칩 내부 논리 회로)

## 메모리 맵 정의

- Gemini 원본
    
    I2C와 WDT(웟치독 타이머)까지 포함하시다니, 이제 정말 상용 칩 수준의 탄탄한 구성이 되었습니다! 무한 루프에 빠져도 스스로 깨어나는 WDT와, 다양한 센서를 주렁주렁 달 수 있는 I2C가 포함된 나만의 RISC-V MCU라니 생각만 해도 가슴이 뜁니다.
    
    본격적인 베릴로그(Verilog) 코딩에 앞서 가장 중요한 설계도인 전체 메모리 맵(Memory Map)을 구체화해 보겠습니다.
    
    SERV 코어는 32비트 주소 체계를 사용하므로 $0x0000\_0000$ 부터 $0xFFFF\_FFFF$ 까지 총 4GB의 광활한 주소 공간을 가집니다. 이 공간을 메모리 영역(Instruction/Data)과 주변장치 영역(Memory-Mapped I/O)으로 깔끔하게 나누는 것이 핵심입니다.

    ### 🗺️ SERV MCU SoC 통합 메모리 맵 (제안)
    
    설계의 편의성(주소 디코딩 단순화)을 위해 각 주변장치에 넉넉하게 4KB(0x1000)씩 주소 공간을 할당하는 방식을 추천합니다.
    
    | **주소 범위 (Address Range)** | **할당된 장치 (Module)** | **크기** | **설명 및 주소 디코딩 힌트** |
    | --- | --- | --- | --- |
    | **`0x0000_0000` ~ `0x00FF_FFFF`** | **External Memory (Pi Pico)** | 16MB | 코어 부팅 시 명령어를 가져오는(Fetch) 메인 메모리 영역. 위시본 ↔ SPI 브리지가 이 주소를 처리합니다. |
    | **`0x4000_0000` ~ `0x4000_0FFF`** | **UART** (시리얼 통신) | 4KB | PC 터미널 통신용 (`[31:12] == 0x40000`) |
    | **`0x4000_1000` ~ `0x4000_1FFF`** | **GPIO** (범용 입출력) | 4KB | LED 켜기, 버튼 읽기 (`[31:12] == 0x40001`) |
    | **`0x4000_2000` ~ `0x4000_2FFF`** | **Timer** (타이머) | 4KB | 딜레이 생성 및 주기적 이벤트 (`[31:12] == 0x40002`) |
    | **`0x4000_3000` ~ `0x4000_3FFF`** | **PWM** (펄스 폭 변조) | 4KB | 모터 제어 및 부저 음향 (`[31:12] == 0x40003`) |
    | **`0x4000_4000` ~ `0x4000_4FFF`** | **SPI** (사용자용) | 4KB | 외부 고속 디스플레이 등 연결 (`[31:12] == 0x40004`) |
    | **`0x4000_5000` ~ `0x4000_5FFF`** | **I2C** (센서 통신) | 4KB | 온습도, 자이로 센서 등 연결 (`[31:12] == 0x40005`) |
    | **`0x4000_6000` ~ `0x4000_6FFF`** | **WDT** (웟치독 타이머) | 4KB | 시스템 강제 리셋 제어 (`[31:12] == 0x40006`) |
    | **`0x4000_7000` ~ `0x4000_7FFF`** | **Interrupt Controller** | 4KB | 모든 주변장치의 인터럽트 통합 관리 (`[31:12] == 0x40007`) |

    ### 🔍 주요 주변장치 내부 레지스터 맵 (예시)
    
    주변장치가 4KB 공간을 할당받았지만, 실제로 그 안에서 쓰는 레지스터(주소)는 몇 개 되지 않습니다. 베릴로그로 설계할 때 이 하위 주소(Offset)들만 매핑해주면 됩니다.
    
    **1. GPIO 레지스터 (Base: `0x4000_1000`)**
    
    - `0x4000_1000` : **GPIO_DATA** (입출력 데이터 값 읽기/쓰기)
    - `0x4000_1004` : **GPIO_DIR** (방향 설정: 1=Output, 0=Input)
    
    **2. UART 레지스터 (Base: `0x4000_0000`)**
    
    - `0x4000_0000` : **UART_DATA** (여기에 값을 쓰면 TX 전송, 읽으면 RX 수신)
    - `0x4000_0004` : **UART_STATUS** (송신 완료 확인, 수신 버퍼 상태 확인 등)
    
    **3. WDT 레지스터 (Base: `0x4000_6000`)**
    
    - `0x4000_6000` : **WDT_FEED** (여기에 특정 비밀번호, 예를 들어 `0xCAFEBABE`를 주기적으로 쓰지 않으면 시스템 리셋)
    - `0x4000_6004` : **WDT_CTRL** (웟치독 활성화 및 타임아웃 시간 설정)

    이렇게 메모리 맵을 짰다는 것은, SERV 코어가 내뱉는 32비트 주소(`wb_adr`)를 보고 데이터가 어느 주변장치로 가야 할지 길을 열어주는 교통경찰(인터커넥트 라우터)이 필요하다는 뜻입니다.
    
    베릴로그 코드로는 대략 이런 느낌이 될 것입니다:
    
    Verilog
    
    # 
    
    `// 간단한 주소 디코딩 로직 예시

    assign sel_uart   = (wb_adr[31:12] == 20'h40000);
    assign sel_gpio   = (wb_adr[31:12] == 20'h40001);`
    
    이제 지도가 완성되었습니다! 코딩을 시작할 준비가 되셨나요?

### 🔍 주요 주변장치 내부 레지스터 맵 (예시)

주변장치가 4KB 공간을 할당받았지만, 실제로 그 안에서 쓰는 레지스터(주소)는 몇 개 되지 않습니다. 베릴로그로 설계할 때 이 하위 주소(Offset)들만 매핑해주면 됩니다.

**1. GPIO 레지스터 (Base: `0x4000_1000`)**

- `0x4000_1000` : **GPIO_DATA** (입출력 데이터 값 읽기/쓰기)
- `0x4000_1004` : **GPIO_DIR** (방향 설정: 1=Output, 0=Input)

**2. UART 레지스터 (Base: `0x4000_0000`)**

- `0x4000_0000` : **UART_DATA** (여기에 값을 쓰면 TX 전송, 읽으면 RX 수신)
- `0x4000_0004` : **UART_STATUS** (송신 완료 확인, 수신 버퍼 상태 확인 등)

**3. WDT 레지스터 (Base: `0x4000_6000`)**

- `0x4000_6000` : **WDT_FEED** (여기에 특정 비밀번호, 예를 들어 `0xCAFEBABE`를 주기적으로 쓰지 않으면 시스템 리셋)
- `0x4000_6004` : **WDT_CTRL** (웟치독 활성화 및 타임아웃 시간 설정)

## 외부에서 가져올 것(IP) /  **우리가 직접 만들 것(Custom Logic)**

하드웨어 시스템 설계에서 "외부에서 가져올 것(IP)"과 **"우리가 직접 만들 것(Custom Logic)"**, 그리고 "그 둘을 연결하는 작업(Integration)"을 명확히 구분하는 것은 전체 프로젝트의 성패를 가르는 가장 훌륭한 엔지니어링 접근법입니다.

하나의 완전한 칩(SoC)을 조립하기 위해 필요한 요소들과 해야 할 일(To-Do)들을 체계적으로 정리해 드립니다.

---

### 📦 1. 오픈소스 및 외부에서 가져와야 하는 것 (Imported IP)

우리가 바퀴를 처음부터 다시 발명할 필요는 없습니다. 이미 검증된 핵심 부품은 그대로 가져다 씁니다.

- **SERV CPU 코어 (RTL 베릴로그 코드):**
    - GitHub에서 SERV 프로젝트를 클론하여 핵심 프로세서 파일(`serv_top.v` 등)을 가져옵니다.
    - 이 코어는 우리가 건드릴 필요가 없는 완벽한 '블랙박스'이자 시스템의 두뇌입니다.
- **ETRI 0.5um 공정 라이브러리 및 패드 프레임:**
    - MPW 주최 측에서 제공하는 물리적 칩의 껍데기(I/O Pad)와 표준 셀(NAND, NOR 게이트 등) 라이브러리 파일입니다. 나중에 합성(Synthesis)할 때 반드시 필요합니다.
- **라즈베리파이 피코 C/C++ SDK:**
    - 외부 메모리 역할을 할 피코의 펌웨어를 작성하기 위한 라즈베리파이 공식 개발 환경입니다.

---

### 🛠️ 2. 우리가 직접 만들어야 하는 모듈 (Custom Logic)

우리 칩만의 개성과 기능을 결정하는 '팔과 다리'입니다. 모두 베릴로그(Verilog)로 직접 코딩해야 합니다.

- **메모리 브리지 (Wishbone ↔ SPI Master):**
    - 가장 중요하고 난이도가 있는 모듈입니다. SERV 코어가 위시본(Wishbone) 버스로 메모리를 요청하면, 이를 가로채서 4가닥의 SPI 신호로 변환해 외부 피코로 보내고 받아오는 로직입니다.
- **주변장치 제어기 (Peripherals):**
    - **UART:** PC와 통신하기 위한 송수신 모듈
    - **GPIO:** LED, 버튼 등을 제어하는 입출력 모듈
    - **Timer & PWM:** 카운터 및 펄스 제어 모듈
    - **I2C / WDT / 인터럽트 제어기:** 칩의 완성도를 높이는 추가 모듈
- *(중요)* **주변장치용 Wishbone 래퍼(Wrapper):**
    - 우리가 만든 UART나 GPIO 모듈이 SERV 코어와 대화하려면 '위시본 버스 프로토콜'이라는 언어를 써야 합니다. 각 주변장치에 위시본 신호를 해석하는 껍데기(Wrapper)를 씌워주어야 합니다.

---

### 🔗 3. 가져온 것과 만든 것을 합치기 위한 작업 (Integration & Glue Logic)

레고 블록(SERV, 주변장치)들을 샀다고 끝이 아닙니다. 이 블록들을 하나의 판(SoC) 위에 조립하고 선을 연결하는 **'글루 로직(Glue Logic)'** 작업이 필요합니다.

- **Wishbone Interconnect (주소 분배기 / 버스 라우터) 설계:**
    - SERV 코어는 하나지만 주변장치는 여러 개입니다. SERV가 내뱉는 주소(`wb_adr`)를 보고, "아, 이건 UART 주소네. UART 모듈 연결해 줘!", "이건 메모리 주소네. SPI 브리지 연결해 줘!" 하고 길을 열어주는 **교통경찰(Multiplexer/Demultiplexer)** 로직을 만들어야 합니다.
- **최상위 모듈 (Top Module) 설계:**
    - `my_mcu_top.v` 라는 가장 바깥쪽 껍데기 파일을 만듭니다.
    - 이 안에 SERV 코어, Interconnect, 모든 주변장치를 부품(Instance)으로 불러와서 내부 선(Wire)들을 서로 엮어줍니다.
    - 마지막으로 이 내부 신호들을 칩 밖으로 나가는 MPW 28핀(I/O Pad)과 물리적으로 연결(Mapping)해 줍니다.
- **소프트웨어 가상 테스트벤치 (C++ / Verilator):**
    - 통합된 하드웨어를 검증하기 위해, PC 환경에서 피코의 메모리 동작을 흉내 내는 C++ 가상 메모리 코드를 작성하여 베릴로그 코드와 융합합니다.
