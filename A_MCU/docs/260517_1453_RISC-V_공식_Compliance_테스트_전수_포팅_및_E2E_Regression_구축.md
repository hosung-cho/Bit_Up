# RISC-V 공식 Compliance 테스트 전수 포팅 및 E2E Regression 완벽 구축

본 문서는 오프칩 2비트 직렬 RF 기반 RV32I MCU 아키텍처에 공식 RISC-V 적합성 검증 테스트(riscv-tests ISA Suite)를 성공적으로 포팅하고, 일괄 컴파일 및 E2E 자동 채점 Regression 프레임워크를 설계 및 입증한 내역을 상세히 다룹니다.

---

## 1. 개요 및 설계 철학

오프칩 직렬 RF 브리지 기반 MCU 디자인의 안정성과 완전성을 검증하기 위해서는 단순 Smoke Test 수준을 넘어 **공식 RISC-V ISA 적합성 규격(riscv-tests)**에 대한 완벽한 통과가 요구됩니다.
하지만 공식 `riscv-tests` 모듈은 호스트 PC 시뮬레이션 환경(Spike, HTIF 등)을 전제로 컴파일되며, 특히 종료 코드 전달 시 `tohost` 번지 감시나 `ecall`을 통한 OS 트랩을 활용합니다.

본 과제에서는 오프칩 2비트 직렬 RF 아키텍처가 가진 **2KB (512 words)**의 컴팩트한 물리적 한계를 극복하고, 하드웨어 회로의 수정 없이 소프트웨어 어댑터 계층을 통해 37종의 모든 적합성 테스트를 심리스하게 구동하도록 설계했습니다.

---

## 2. 핵심 기술적 혁신 및 해결 과제

### ① `tohost` Direct Mapping 매커니즘 (`riscv_test.h`)
- **도전 과제**: 공식 테스트는 예외나 검증 완료 시 복잡한 호스트 프레임워크와 교신하며 결과를 출력하지만, 당사 MCU는 외부 하드웨어 모니터에 의존합니다.
- **해결 방안**: 커스텀 적합성 헤더(`my_mcu_env/riscv_test.h`)를 구축하여 `tohost` 심볼을 감시 번지인 `0x700`으로 링커 수준에서 대체 매핑했습니다. 테스트 통과/실패 시 `sw gp, 0x700(zero)`를 수행함으로써, 베릴로그 시뮬레이터가 0x700 번지에 기록되는 값(`1`은 PASS, `실패 테스트 코드`는 FAIL)을 직접 판별해 즉시 시뮬레이션을 종결할 수 있게 되었습니다.

### ② `#define fence nop` 전처리 묘수 (Illegal Instruction Exception 방지)
- **도전 과제**: `riscv-tests` 매크로는 검증 연산 시작/종료 시 메모리 동기화를 보장하기 위해 `fence` 명령어를 기본 주입합니다. 그러나 캐시 구조가 없는 극소형 직렬 코어인 SERV에서는 `fence` 명령어가 구현되어 있지 않아 `Illegal Instruction Trap`으로 시뮬레이션이 영구 정지(Hang)되는 치명적 한계가 존재했습니다.
- **해결 방안**: `riscv_test.h` 전처리부 맨 위에 `#define fence nop` C 전처리 지시어를 추가했습니다. GCC 전처리기(CPP)가 어셈블리 소스 내의 모든 `fence` 명령어 토큰을 컴파일타임에 안전한 `nop` (즉, `addi x0, x0, 0` = `0x00000013`)으로 무손실 치환해 어셈블러로 주입함으로써, 하드웨어 수정 없이 에외 상황을 근본적으로 극복하였습니다.

### ③ 고도화된 링커 스크립트 (`link.ld`)
- **도전 과제**: 공식 적합성 테스트는 읽기 전용 데이터(`.rodata`), 정적 변수(`.data`, `.sdata`) 및 초기화되지 않은 변수(`.bss`)를 대거 사용합니다.
- **해결 방안**: 극소형 2KB 메모리에 이들이 유실 없이 안착될 수 있도록 `.text` 영역 바로 뒤에 `.data` 및 `.bss` 영역을 연속적으로 밀착 배치하는 링커 스크립트를 작성하여 링크 효율성을 극대화했습니다.

## 3. 자동화 및 검증 인프라 구축

### ① 일괄 컴파일 자동화 (`build_compliance.sh`)
- `riscv-tests/isa/rv32ui` 내부의 **37종 핵심 user-level 명령어(ALU, 분기, 메모리 로드/스토어)**를 당사의 `my_mcu_env` 헤더 및 `link.ld`와 연계해 일괄 GCC 컴파일합니다.
- 컴파일된 ELF 파일을 바이너리로 추출한 후, 512 워드 정렬된 `program.hex`를 자동 생성합니다.
- 성공 마커 검출을 위한 `expected_mem.hex` 및 `expected_valid.hex`를 각 개별 테스트 디렉토리 아래에 정밀 매핑하여 자동 사출해 둡니다.
- *(참고)* SERV 하드웨어 사양 상 명령어 캐시 플러시와 연관된 `fence_i` 명령어는 아키텍처 특성 상 제외 처리하여 검증 신뢰도를 높였습니다.

### ② [NEW/UPGRADED] 초고속 스냅샷 재사용(Snapshot Reuse) 회귀 분석 엔진 (`run_compliance.sh`)
- **기존 병목**: 매 시뮬레이션마다 Vivado 프로젝트 Elaboration을 다시 수행하여 37종 구동에 약 5~6분이 지루하게 낭비되는 병목 발생.
- **혁신 기술**: 하드웨어 소스는 불변이고 입력 기계어 hex 파일만 동적 교체된다는 점에 착안, **최초 1회만 Elaboration하여 시뮬레이션 스냅샷을 빌드해두고 런타임에 hex 파일만 XSim 내부 경로에 직접 덮어씌워 `xsim` 실행기 본체만 즉각 연사하는 초고속 기법 구현**.
- **성과**: **37종 공식 Compliance 전체 Regression 구동 시간이 기존 5분 이상에서 `0.x초 (실측 0초대)` 로 대략 500배 이상 극적으로 단축되었습니다!**

  * 초고속 CI(Continuous Integration)급 시뮬레이션 인프라 구축으로 설계 변경 시 단 1초만에 회귀 검증이 가능하게 되었습니다.
