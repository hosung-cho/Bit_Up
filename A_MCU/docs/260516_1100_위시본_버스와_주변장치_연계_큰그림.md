# Wishbone 버스와 주변장치 연계 큰 그림

이 문서는 SERV 기반 SoC에서 오프칩 메모리를 설계하기 전에 먼저 잡아야 할 전체 구조를 정리한다. 핵심은 명령어 메모리, 데이터 메모리, 주변장치, 레지스터 파일이 모두 같은 방식으로 붙는 것이 아니라는 점이다.

```text
                 +------------------+
                 |     SERV CPU     |
                 |                  |
                 |  ibus            |----> 명령어 메모리 경로
                 |  dbus            |----> 데이터 메모리 / 주변장치 경로
                 |  rf interface    |----> 레지스터 파일 경로
                 +------------------+
```

역할은 다음처럼 나누어 보는 것이 좋다.

```text
ibus : instruction fetch 전용
dbus : load/store 전용
rf   : CPU 레지스터 파일 전용
```

명령어 메모리, 데이터 메모리, UART, GPIO, Timer 같은 주변장치는 Wishbone 쪽에서 생각한다. 레지스터 파일은 Wishbone이 아니라 SERV 전용 RF interface에서 따로 다룬다.

## Wishbone 버스의 역할

Wishbone은 CPU 같은 master와 메모리/주변장치 같은 slave 사이의 공통 통신 규약이다. CPU가 하고 싶은 일은 결국 두 가지다.

```text
주소 A에서 읽기
주소 B에 데이터 D 쓰기
```

SERV의 데이터 버스는 다음 신호들로 구성되어 있다.

```verilog
o_dbus_adr
o_dbus_dat
o_dbus_sel
o_dbus_we
o_dbus_cyc
i_dbus_rdt
i_dbus_ack
```

각 신호의 의미는 다음과 같다.

| 신호    | 의미                                         |
| ------- | -------------------------------------------- |
| `adr` | 접근 주소                                    |
| `dat` | CPU에서 slave로 나가는 쓰기 데이터           |
| `rdt` | slave에서 CPU로 들어오는 읽기 데이터         |
| `we`  | write enable.`1`이면 write, `0`이면 read |
| `sel` | byte enable. byte/halfword/word 접근에 사용  |
| `cyc` | 요청 valid                                   |
| `ack` | slave 응답 완료                              |

읽기 흐름은 다음과 같다.

```text
CPU:    adr=0x00000010, we=0, cyc=1
Slave:  rdt=읽은값, ack=1
CPU:    ack를 보고 읽기 완료
```

쓰기 흐름은 다음과 같다.

```text
CPU:    adr=0x00000010, dat=0x12345678, sel=1111, we=1, cyc=1
Slave:  데이터를 저장하고 ack=1
CPU:    ack를 보고 쓰기 완료
```

`ack`가 중요하다. 온칩 RAM처럼 빠른 slave는 바로 응답할 수 있고, SPI RAM/Flash 같은 오프칩 메모리는 여러 사이클 뒤에 응답해도 된다. CPU는 `ack`가 올 때까지 기다린다.

이 프로젝트의 SERV 포트에는 일반 Wishbone에서 흔히 보이는 `stb`가 없다. 따라서 단순화된 Wishbone 스타일로 보고, 필요하면 bridge 내부에서 `stb =`

` cyc`처럼 취급하면 된다.

## ibus와 dbus

SERV에는 Wishbone 스타일 버스가 두 개 있다.

```text
ibus : 명령어 fetch
dbus : load/store
```

`ibus`는 PC 주소로 다음 실행할 명령어를 읽는다.

```text
PC 주소 -> instruction memory -> instruction word
```

`dbus`는 프로그램의 load/store 명령이 사용한다.

```c
x = *(uint32_t *)0x00000100;       // dbus read
*(uint32_t *)0x40000000 = 0x55;    // dbus write
```

기본 구조는 다음과 같다.

```text
SERV CPU
 ├─ ibus -> instruction memory
 └─ dbus -> data memory / UART / GPIO / timer / etc
```

처음 설계할 때는 `ibus`와 `dbus`를 분리해서 생각하는 것이 쉽다.

```text
ibus -> ROM / Flash / instruction RAM
dbus -> RAM + peripherals
```

나중에 instruction memory와 data memory를 물리적으로 같은 오프칩 메모리에 넣고 싶다면 arbiter가 필요하다.

```text
        ibus ----+
                 +-- arbiter --> single off-chip memory
        dbus ----+
```

arbiter는 `ibus`와 `dbus`가 동시에 요청할 때 어느 쪽을 먼저 처리할지 정하는 블록이다.

## 주소 맵

SoC에서는 모든 장치를 주소 공간에 배치한다. 이것을 memory map 또는 address map이라고 한다.

예시는 다음과 같다.

```text
0x0000_0000 - 0x0000_FFFF : RAM / instruction + data memory
0x1000_0000 - 0x100F_FFFF : external flash
0x4000_0000 - 0x4000_00FF : UART
0x4000_0100 - 0x4000_01FF : GPIO
0x4000_0200 - 0x4000_02FF : Timer
0x4000_0300 - 0x4000_03FF : SPI controller
```

CPU는 주변장치를 특별한 명령으로 접근하지 않는다. 그냥 주소로 load/store 한다.

```c
*(volatile uint32_t *)0x40000000 = 'A';
```

그러면 하드웨어 주소 디코더가 주소를 보고 UART를 선택한다.

현재 `my_mcu_top.v`에도 같은 방향의 구조가 들어가 있다.

```verilog
wire sel_mem  = (wb_dbus_adr[31:28] == 4'h0); // 0x0000_XXXX
wire sel_peri = (wb_dbus_adr[31:28] == 4'h4); // 0x4000_XXXX

wire sel_uart = sel_peri && (wb_dbus_adr[11:8] == 4'h0); // 0x4000_00XX
wire sel_gpio = sel_peri && (wb_dbus_adr[11:8] == 4'h1); // 0x4000_01XX
```

즉 의도는 다음과 같다.

```text
0x0000_xxxx : memory
0x4000_00xx : UART
0x4000_01xx : GPIO
```

## 주소 디코더와 MUX

데이터 버스 쪽 interconnect는 보통 다음처럼 생긴다.

```text
                  +----------------+
SERV dbus ------> | Address Decode |
                  +----------------+
                    |    |    |
                    |    |    +--> GPIO
                    |    +-------> UART
                    +------------> RAM

각 slave의 rdata/ack ----> MUX ----> SERV dbus rdt/ack
```

Verilog 개념은 다음과 같다.

```verilog
assign mem_cyc  = wb_dbus_cyc && sel_mem;
assign uart_cyc = wb_dbus_cyc && sel_uart;
assign gpio_cyc = wb_dbus_cyc && sel_gpio;

assign wb_dbus_rdt =
    sel_mem  ? mem_rdt  :
    sel_uart ? uart_rdt :
    sel_gpio ? gpio_rdt :
               32'h0;

assign wb_dbus_ack =
    sel_mem  ? mem_ack  :
    sel_uart ? uart_ack :
    sel_gpio ? gpio_ack :
               1'b0;
```

CPU는 하나의 `dbus`만 가지고 있고, 주소 디코더가 그 요청을 여러 slave 중 하나로 보내준다.

## 주변장치 연결 방식

주변장치는 대부분 "레지스터 몇 개"처럼 보이게 만든다. 이것을 MMIO, memory-mapped I/O라고 한다.

UART 주소 맵 예시는 다음과 같다.

```text
0x4000_0000 : UART TXDATA
0x4000_0004 : UART RXDATA
0x4000_0008 : UART STATUS
0x4000_000C : UART CTRL
```

CPU 프로그램에서는 메모리 접근처럼 쓴다.

```c
#define UART_TXDATA (*(volatile unsigned int *)0x40000000)
#define UART_STATUS (*(volatile unsigned int *)0x40000008)

UART_TXDATA = 'A';
```

하드웨어에서는 주소 하위 비트를 보고 어느 레지스터 접근인지 고른다.

```verilog
case (wb_adr[3:2])
  2'd0: txdata register
  2'd1: rxdata register
  2'd2: status register
  2'd3: ctrl register
endcase
```

GPIO도 같은 방식으로 붙일 수 있다.

```text
0x4000_0100 : GPIO_OUT
0x4000_0104 : GPIO_IN
0x4000_0108 : GPIO_DIR
```

Timer도 마찬가지다.

```text
0x4000_0200 : TIMER_COUNT
0x4000_0204 : TIMER_COMPARE
0x4000_0208 : TIMER_CTRL
```

버스 관점에서는 RAM과 주변장치 모두 slave다. 차이는 내부 동작뿐이다.

```text
RAM   : 주소에 따라 데이터를 저장/읽음
UART  : 특정 주소에 쓰면 송신 시작
GPIO  : 특정 주소에 쓰면 핀 출력 변경
Timer : 특정 주소를 읽으면 카운터 값 반환
```

## 오프칩 메모리 bridge

오프칩 메모리는 Wishbone slave처럼 보이게 만들면 된다.

```text
SERV ibus/dbus
   |
   v
Wishbone-to-Offchip-Memory Bridge
   |
   v
SPI / QSPI / parallel SRAM / external MCU / etc
```

bridge가 해야 할 일은 다음과 같다.

```text
1. Wishbone 요청 받기
2. 주소, read/write, byte enable 해석
3. 오프칩 프로토콜 수행
4. 완료되면 rdt와 ack 반환
```

SPI RAM read 예시는 다음과 같다.

```text
CPU dbus read 요청
  -> bridge가 SPI command 전송
  -> address 전송
  -> data 수신
  -> wb_rdt에 넣음
  -> wb_ack=1
```

CPU는 `ack`가 올 때까지 기다리므로, 오프칩 메모리가 느려도 프로토콜만 정확하면 동작할 수 있다. 다만 성능은 낮아진다.

## 레지스터 파일은 별도 경로

레지스터 파일은 Wishbone으로 빼는 것이 아니라 SERV 전용 RF interface로 다루는 것이 맞다.

```text
SERV
 ├─ ibus/dbus  -> instruction/data memory, peripheral
 └─ RF port    -> register file storage
```

현재 구조는 다음 방향으로 잡혀 있다.

```text
SERV RF port
    |
    v
serv_rf_ram_if
    |
    v
serv_rf_ram 또는 external RF bridge
    |
    v
off-chip register file storage
```

메모리와 주변장치는 Wishbone 주소 맵에 배치하고, RF는 CPU 내부 실행 타이밍에 직접 걸리는 별도 저장소로 본다. RF는 `rf_ready`와 read data valid 타이밍이 중요하다.

## 추천 전체 구조

처음에는 다음 구조로 잡는 것이 가장 이해하기 쉽다.

```text
                         +-------------------+
                         |      SERV CPU     |
                         +-------------------+
                           |       |       |
                         ibus    dbus     rf
                           |       |       |
                           |       |       +--> RF bridge -> off-chip register file
                           |       |
                           |       v
                           |   +-------------------+
                           |   |  Address Decoder  |
                           |   +-------------------+
                           |     |      |      |
                           |     |      |      +--> GPIO slave
                           |     |      +---------> UART slave
                           |     +----------------> data memory bridge
                           |
                           v
                   instruction memory bridge
```

추천 주소 맵 초안은 다음과 같다.

```text
0x0000_0000 - 0x0000_FFFF : instruction/data RAM
0x1000_0000 - 0x100F_FFFF : external flash
0x4000_0000 - 0x4000_00FF : UART
0x4000_0100 - 0x4000_01FF : GPIO
0x4000_0200 - 0x4000_02FF : Timer
0x4000_0300 - 0x4000_03FF : SPI controller
```

`RESET_PC`가 `0x0000_0000`이면 부팅 시 `ibus`가 `0x0000_0000`에서 첫 명령어를 가져온다. 따라서 그 주소에는 instruction memory가 있어야 한다.

## 설계 순서

권장 순서는 다음과 같다.

1. 주소 맵 확정
2. 온칩 더미 RAM으로 `ibus`/`dbus` 검증
3. UART/GPIO 같은 간단한 MMIO slave 추가
4. 주소 디코더와 `rdata`/`ack` MUX 검증
5. 오프칩 memory bridge 추가
6. `ibus`와 `dbus`가 같은 오프칩 메모리를 공유할지 결정
7. 공유한다면 arbiter 추가
8. RF off-chip 경로는 별도로 타이밍 검증

## 핵심 요약

```text
Wishbone = CPU가 메모리/주변장치에 load/store 하는 공통 버스
주소 디코더 = 주소를 보고 RAM/UART/GPIO 중 누구에게 보낼지 결정
MMIO = 주변장치를 메모리 주소처럼 보이게 만드는 방식
오프칩 메모리 bridge = Wishbone 요청을 SPI/SRAM/외부 프로토콜로 변환
RF = Wishbone이 아니라 SERV 전용 RF interface로 따로 처리
```

따라서 이 SoC의 큰 구조는 다음처럼 이해하면 된다.

```text
SERV + ibus instruction path + dbus MMIO interconnect + RF off-chip path
```
