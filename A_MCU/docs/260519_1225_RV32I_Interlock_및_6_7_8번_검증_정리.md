# RV32I Off-chip RF Interlock 및 6/7/8번 검증 정리

작성 시각: 2026-05-19 12:25 KST

## 목표

- 1mm x 1mm MPW tape-out을 우선 목표로 하되, CPU는 느려도 정확하게 동작해야 한다.
- 원본 SERV core 내부 파일은 보존한다.
- off-chip RF 구조에서 직전 writeback 값이 다음 명령의 RF read에 늦게 반영되는 문제를 top-level/adapter 계층에서 해결한다.

## RTL 진행 상태

### 1. 정확성 우선 instruction interlock

`src/rtl/my_mcu_top.v`에서 instruction ack를 RF write drain 이후로 늦추는 방식으로 interlock을 적용했다.

- SERV가 다음 instruction을 fetch하더라도, 이전 instruction의 GPR write burst가 off-chip RF에 끝까지 배출되기 전에는 CPU-facing `wb_ibus_ack`를 보류한다.
- 성능은 낮아지지만 RF RAW hazard가 사라지는 방향이다.
- SERV core 내부 RTL은 수정하지 않았다.

### 2. CSR-off rs2 decode 오류 수정

`WITH_CSR=0`일 때 일반 레지스터 x16-x31을 CSR 주소처럼 취급하던 조건을 막았다.

- 기존 문제: `rreg1[4+WITH_CSR]` 조건이 `WITH_CSR=0`에서도 x16 이상을 CSR 경로처럼 해석할 수 있었다.
- 결과: RV32I 일반 레지스터 read가 정상화되었다.

### 3. RF writeback phase hint 최소화

기존 32-bit `current_insn` shadow를 제거하고, 필요한 instruction class만 1-bit hint로 보존했다.

- 대상: boolean ALU(`xor/or/and` 계열), `sub`, U-type(`lui/auipc`)
- 목적: off-chip RF 2-bit packing phase를 맞추되 면적 증가를 줄인다.
- 합성 전용 관점에서 32-bit shadow보다 훨씬 작다.

### 8번 Dhrystone 테스트

초기 실패 원인은 RTL이 아니라 테스트벤치/프로그램 조건이었다.

수정 내용:

- `programs/dhrystone.c`
  - 기존 `Arr_1_Glob[50]`, `Arr_2_Glob[50][50]` 때문에 `.bss`가 2KB 외부 RAM을 초과했다.
  - 실제 접근 범위인 `Arr_1_Glob[8]`, `Arr_2_Glob[8][7]`만 담도록 `Arr_1_Glob[9]`, `Arr_2_Glob[9][8]`로 축소했다.
- `tb_rv32i_dhrystone.v`
  - 16MHz fast clock, `CLK_SYS_DIV=32` 조건에서 reset 200ns는 너무 짧았다.
  - reset을 20us로 늘려 reset 상태에서 `clk_sys` edge를 충분히 받게 했다.
  - timeout 시 PC, instruction, RF/memory frame 요약을 출력하도록 진단 로그를 추가했다.

최종 상태: PASS

## 합성 면적 상태

해석:

- Yosys cell 면적 기준으로는 1mm x 1mm 미만이다.
- 다만 routing/pad/power 여유까지 고려하면 아직 매우 타이트하다.
- 다음 단계는 실제 P&R 또는 floorplan 기반 utilization 확인이 필요하다.

## 현재 결론

- CSR 제외 RV32I 경로는 5/6/7/8번 테스트 기준으로 정상 동작한다.
- off-chip RF RAW hazard는 instruction interlock + 최소 phase hint로 해결된 상태다.
- 원본 SERV core 내부 수정 없이 top-level 계층에서 수습했다.
- 남은 핵심 리스크는 RTL 기능보다 physical implementation 여유다.
