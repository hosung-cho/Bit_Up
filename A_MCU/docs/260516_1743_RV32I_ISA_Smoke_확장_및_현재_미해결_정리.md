# RV32I ISA Smoke 확장 및 현재 미해결 정리

작성 시각: 2026-05-16 17:43 KST

## 질문에 대한 답

현재 상태에서 "RISC-V 기본 ISA가 전부 실제로 돈다"고 말할 수는 없다.

기존 top-level smoke는 다음 범위를 확인했다.

- `addi`
- `lw/lb/lbu/lh/lhu`
- `sw/sb/sh`
- `beq/bne`
- `jal/jalr`
- CSR shadow `mscratch/mtvec/mepc/mtval`

하지만 RV32I 전체 관점에서는 아직 부족했다. 특히 `lui/auipc`, R-type 산술/논리/시프트, I-type 비교/논리/시프트, 전체 branch 6종을 한 번에 묶은 테스트가 없었다.

## 이번에 추가한 테스트

파일: `testbench/4_testbench_my_mcu_top/tb_my_mcu_top.v`

기존 smoke 프로그램 뒤에서 `jal x0, 0x200`으로 별도 ISA smoke 영역에 진입하도록 했다.

추가한 명령 범위:

- U-type: `lui`, `auipc`
- R-type: `add`, `sub`, `sll`, `slt`, `sltu`, `xor`, `srl`, `sra`, `or`, `and`
- I-type ALU: `slti`, `sltiu`, `xori`, `ori`, `andi`, `slli`, `srli`, `srai`
- Branch: `beq`, `bne`, `blt`, `bge`, `bltu`, `bgeu`

각 결과는 `0x180..0x1e4` 결과 메모리 영역에 `sw`로 남기고 테스트벤치가 기대값과 비교한다.

- load/store/GPIO smoke 통과
- branch/JAL/JALR 기존 smoke 통과
- CSR shadow `rf32..rf35` 통과

하지만 새 RV32I 확장 smoke는 실패한다.

대표 실패:

- `rf5`
  - observed `14000000`
  - expected `00000005`
- `rf8`
  - observed `e3ffffff`
  - expected `fffffff8`
- `lui -> store`
  - observed `2468a000`
  - expected `12345000`
- R-type/I-type 결과 저장 영역 다수 실패

중간에 시도한 수정:

- `rd_mem_en`을 `serv_top` 밖으로 노출
- `load`를 제외한 RD writeback에 next phase를 적용하는 실험

결과:

- `lui -> store` 일부는 개선됐지만 `jal/jalr` link 값이 깨졌다.
- 따라서 그 방식은 정식 해법이 아니어서 `i_wdata0_next` 연결은 기존 `rd_alu_en | rd_csr_en`으로 되돌렸다.
- `o_rd_mem_en` 노출 wiring은 남아 있지만 현재 phase 선택에는 사용하지 않는다.

## 현재 판단

문제는 코어의 RV32I decode 자체라기보다는, 오프칩 RF adapter에서 명령 종류별 RD writeback phase가 아직 충분히 일반화되지 않은 쪽에 가깝다.

지금까지는 `addi`, load, CSR, jump link 같은 일부 대표 경로만 맞춰 왔다. RV32I 전체 smoke를 넣자 다음 계열의 phase 차이가 추가로 드러났다.

- U-type writeback: `lui/auipc`
- R-type ALU writeback
- 일부 I-type ALU writeback
- 그 결과를 다시 store source로 읽는 경로

## 다음 작업

1. `serv_decode`의 writeback source 분류를 더 세밀하게 밖으로 노출한다.
   - `rd_alu_en`
   - `rd_csr_en`
   - `rd_mem_en`
   - `utype`
   - `jal_or_jalr`
   - 필요하면 `two_stage_op`/shift 계열
2. RF adapter의 `i_wdata0_next`를 단일 boolean이 아니라 source별 phase 선택으로 정리한다.
3. 작은 단위부터 다시 닫는다.
   - `addi x5, x0, 5`
   - `lui x7, ...`
   - `add x7, x5, x6`
   - `sw x7, ...`
4. 그 후 현재 추가한 RV32I smoke 전체를 다시 통과시킨다.

## 결론

현재 답은 명확하다.

- 기존 smoke 기준의 MCU 기능은 꽤 많이 통과했다.
- 하지만 RV32I 기본 ISA 전체가 오프칩 RF/메모리 구조에서 전부 돈다는 검증은 아직 통과하지 않았다.
- 이번 작업으로 그 사실을 드러내는 top-level ISA smoke가 추가됐고, 다음 핵심 과제는 writeback phase 분류를 명령 source별로 정식화하는 것이다.

## 2026-05-16 21:00 KST 추가 진행

추가로 RF write burst의 chunk 주소가 이전 `rcnt` 상태에 끌려가던 문제를 확인했다.

증상:

- `addi x5, x0, 5`의 write frame이 `chunk=13,14,15,0,...` 순서로 나가며 `rf5=0x14000000`처럼 기록됨

수정:

- `serv_rf_ram_if.v`에 write burst 전용 `write_chunk` counter 추가
- `i_wreq`마다 `write_chunk=0`으로 리셋
- port0/port1 한 쌍의 write slot이 지나갈 때 chunk 증가

효과:

- `rf5`, `rf6`, `rf8`, `rf9` 직접 writeback 정상화
- RV32I smoke 실패 수가 크게 줄어듦
- `add/sub/xor/or/and`, branch 6종 결과가 통과
- `testbench/3_testbench_rf_if_serial/sim/./run.sh` 통과 유지

현재 남은 실패:

- `lui`
- `auipc`
- `sll/slli`
- `srl/srli`
- `sra/srai`
- `slt/sltu/slti/sltiu`

즉 남은 문제는 RF write burst chunk 주소가 아니라, U-type/shift/compare 계열의 RD writeback phase 분류다.

## 2026-05-16 22:05 KST 추가 확인 - 직접 mepc rewrite 재실험

`mepc = mepc + 4` read-modify-write가 아니라, handler에서 상수 `0x4bc`를 직접 `mepc`에 쓰는 더 단순한 실험을 진행했다.

실험 흐름:

1. `ecall` trap 진입
2. handler에서 `addi x2, x0, 0x4bc`
3. `sw x2, 0x1f4(x0)`로 상수 생성 확인
4. `csrrw x3, mepc, x2`
5. 충분한 NOP 후 `mret`

관찰:

- `memory[0x1f4] = 000004bc`로 `x2` 상수 생성은 정상
- RF34(`mepc`) write trace에서 `csrrw mepc,x2`가 실제로 `0x4bc` chunk를 쓰는 것까지 확인
- 그럼에도 바로 뒤 `mret`은 `0x4bc`를 안정적으로 PC로 사용하지 못함
- 이후 데이터 영역 instruction fetch/trap 및 앞쪽 smoke 재실행 흔적이 나타남

해석 업데이트:

- `mepc` software rewrite 실패 원인은 `csrrw` write 자체가 빠지는 문제가 아니다.
- 더 정확히는 "CSR shadow RF34에 write된 직후, 다음 `mret`의 read-side/prefetch/PC 소비 순서가 안정적으로 닫히지 않는 문제"에 가깝다.
- 즉 남은 핵심은 port1 CSR write data phase보다, CSR shadow write 완료와 `mret` read request 사이의 ordering/ready 조건이다.

이번 실험은 실패 케이스라 테스트벤치에는 남기지 않았다. 현재 유지한 top-level smoke는 기존처럼 `ecall` instruction을 `nop`으로 패치한 뒤 `mret` 복귀를 확인하는 통과 버전이다.

## 2026-05-16 21:40 KST 추가 진행 - ecall/mret smoke 추가

기본 RV32I smoke 뒤에 trap/mret smoke를 추가했다.

추가한 흐름:

1. ISA smoke 종료 직후 `mtvec=0x130`으로 설정
2. `ecall` 실행
3. `0x130` handler로 trap 진입
4. handler에서 `mcause`를 `0x1e8`에 저장
5. handler가 `ecall` instruction word를 `nop(0x00000013)`으로 패치
6. `mret` 실행
7. 원래 `mepc`인 `ecall` 주소로 복귀하되, 해당 instruction이 `nop`으로 바뀌었으므로 다음 instruction으로 진행
8. 복귀 marker를 `0x1f0`에 저장

검증된 값:

- `memory[0x1e8] = 0000000b`
  - `ecall` exception code 11
- `memory[0x1f0] = 00000001`
  - `mret` 이후 정상 복귀 marker
- 최종 PC는 halt loop `0x4d4`에 도달

이번에 수정한 RTL:

- `my_mcu_top.v`
  - RF read predecode에서 SYSTEM opcode(`1110011`)는 일반 rs2 predecode로 덮지 않도록 수정
  - 기존에는 `funct3!=0`인 CSR op만 보호했기 때문에 `ecall/mret` 같은 `funct3=0` system op에서 trap/mret용 `mtvec/mepc` read가 rs2 경로에 덮일 수 있었다.

검증 결과:

결과:

- `[TB PASS] my_mcu_top executed off-chip instruction/data memory program`


현재 남은 점:

- `ecall` trap 진입과 `mret` 복귀는 top-level smoke에서 확인됐다.
- handler에서 `mepc`를 software로 `mepc+4`로 다시 쓰는 경로는 아직 정식 해결하지 않았다.
- 이번 smoke는 `mepc` rewrite 대신 instruction memory의 `ecall` word를 `nop`으로 패치해서 `mret` 복귀 자체를 검증한다.
- 다음 보강 후보:
  - `mepc` CSR write/read timing 정식화
  - `ebreak` exception code 3
  - misaligned load/store/jump trap
  - timer interrupt
  - riscv-tests/riscv-formal 기반 compliance

## 2026-05-16 21:49 KST 추가 확인 - mepc rewrite/ebreak 실험 결과

`mepc` software rewrite를 다시 시도했다.

실험 흐름:

1. handler에서 `csrrs x2, mepc, x0`
2. `x2`를 `0x1ec`에 저장
3. `addi x2, x2, 4`
4. `x2`를 `0x1f4`에 저장
5. `csrrw x3, mepc, x2`
6. 충분한 NOP 후 `mret`

관찰:

- `x2 = mepc + 4 = 0x4bc` 계산 자체는 확인됨
- 그러나 `mepc` shadow slot이 안정적으로 `0x4bc`로 갱신되지 않음
- `mret` 복귀 marker가 찍히지 않거나, 프로그램이 다시 앞쪽 smoke를 도는 흔적이 나타남

해석:

- ALU 계산 문제는 아니다.
- `mepc` CSR write/read timing, 특히 off-chip RF의 CSR shadow writeback phase와 바로 뒤 `mret`의 CSR PC read 사이가 아직 정식화되지 않았다.
- 따라서 현재 본 테스트에는 `mepc+4` rewrite 경로를 넣지 않았다.

`ebreak`도 별도 handler로 실험했다.

실험 흐름:

1. `mtvec=0x600`
2. `ebreak`
3. handler에서 `mcause`를 `0x1f4`에 저장
4. `ebreak` instruction word를 `nop`으로 패치
5. `mret`

관찰:

- handler 진입 성공
- `memory[0x1f4] = 00000003`으로 `ebreak` exception code 3 저장 확인
- 그러나 `mret` 이후 복귀 marker까지는 안정적으로 닫히지 않음

현재 유지한 통과 범위:

- RV32I 주요 명령 smoke

- off-chip RF serial bridge
- CSR shadow 기본 read/write
- `ecall` exception code 11
- `ecall` handler 진입
- `mret` 복귀 smoke

현재 제외한 범위:

- handler에서 `mepc = mepc + 4`로 software rewrite 후 `mret`
- `ebreak` 이후 `mret` marker까지의 완전 복귀
