## 2. 모듈별 면적 급증 원인 분석

### ① `serv_rf_ram_if` (Area: 264,213)
* **원인**: 
  1. 외부 피코로부터 오프칩 RF 데이터(32비트 피드 2개 = 64비트)를 2비트 폭으로 prefetch하여 보관하는 64비트의 레지스터 버퍼(`read_buf0`, `read_buf1`)를 가집니다.
  2. SERV 코어에 매 사이클마다 데이터를 1비트씩 공급할 때, `stream_cnt` (5비트 레지스터)를 인덱스로 사용하여 **`read_buf0[stream_cnt]`** 및 **`read_buf1[stream_cnt]`**와 같이 **동적 인덱싱(dynamic indexing)**을 수행합니다.
  3. 이 동적 인덱싱으로 인해 Yosys가 거대한 **32:1 MUX 2개**와 이와 관련된 복잡한 디코더 조합 회로를 생성하여 면적이 대폭 증가했습니다. (DFF 면적은 약 10만이나, 조합 회로 면적이 16만 이상을 차지)

### ② `offchip_mem_bridge` (Area: 132,606)
* **원인**:
  1. 외부 메모리 통신 프레임(`active_frame`)은 총 70비트(WE, IBUS, SEL, ADDR, DATA)로 구성됩니다.
  2. 이 프레임을 1비트 직렬 신호(`o_mem_mosi`)로 변환하여 출력하기 위해 `active_frame[bit_count - 1'b1]` 구조를 사용합니다.
  3. 7비트 카운터 `bit_count`로 70비트 버스 신호를 선택하는 과정에서 거대한 **70:1 MUX 트리 조합 회로**가 합성되어 면적을 크게 낭비하고 있습니다.

### ③ `my_mcu_top` (Area: 136,512)
* **원인**:
  1. 오프칩 RF의 느린 직렬 통신 속도로 인해, 이전 명령어의 GPR 쓰기가 완료되기 전에 다음 명령어 fetch가 끝날 경우 데이터 해저드(stale operand 읽기)가 발생합니다.
  2. 이를 방지하고자 이미 로드된 다음 명령어를 잠시 보관하는 **32비트 임시 명령어 버퍼 `ibus_pending_rdt`**와 관련 인터록 제어 레지스터들을 가지고 있습니다.
  3. 이 레지스터들이 비동기 리셋 구조로 선언되어 면적이 매우 큰 **비동기 리셋 플립플롭(`DFFSR`)이 46개나 생성**되어 local 면적의 53% 이상을 점유하고 있습니다.

## 3. 면적 최소화를 위한 혁신적인 최적화 제안 (Area Diet)

위의 분석 결과를 바탕으로, 기능적 무결성을 유지하면서 물리 면적을 최소 30% 이상 절감할 수 있는 3가지 아키텍처 최적화 방안을 제안합니다.

### 💡 제안 1: `serv_rf_ram_if` 버퍼를 Shift Register 구조로 변경 (조합회로 대폭 감축)
* **AS-IS**: `read_buf0[stream_cnt]` 처럼 동적 MUX를 이용해 특정 비트를 선택하여 출력.
* **TO-BE**: `read_buf0`과 `read_buf1`을 **우측 쉬프트 레지스터(Right Shift Register)** 형태로 재설계합니다.
  * 외부 prefetch 데이터를 채워 넣을 때도 2비트씩 순차적으로 shift-in 시킵니다. (`read_buf0 <= {i_rdata, read_buf0[31:2]}`)
  * SERV 코어가 데이터를 소비할 때도 매 클럭마다 1비트씩 우측 쉬프트 시키고, 출력은 항상 고정된 LSB 비트(`read_buf0[0]`, `read_buf1[0]`)만 사용합니다.
* **효과**: 32:1 MUX 2개와 복잡한 인덱싱 디코더가 완벽히 제거되어, DFF 개수는 동일하게 유지되면서 조합 회로 면적이 극적으로 축소됩니다. (약 8~10만 Area 절감 예상)

### 💡 제안 2: `offchip_mem_bridge` 32비트 Shift Register 재사용 구조 도입 (70:1 MUX 제거)
* **AS-IS**: 70비트 프레임 신호에 대해 `bit_count` 기반 70:1 MUX를 사용하여 1비트씩 출력.
* **TO-BE**: 70비트 전체를 별도로 레지스터링하거나 거대한 MUX를 쓰는 대신, **32비트짜리 소형 Shift Register 하나만 사용**하도록 시리얼 전송 구조를 단계화합니다.
  * **Phase 0 (첫 6비트)**: WE, IBUS, SEL 신호를 아주 작은 조합 회로 MUX(6:1 MUX)를 통해 직접 전송.
  * **Phase 1 (다음 32비트)**: 32비트 Address를 Shift Register에 적재한 후 1비트씩 shift-out 전송.
  * **Phase 2 (마지막 32비트)**: 32비트 Data를 동일한 Shift Register에 재적재한 후 1비트씩 shift-out 전송.
* **효과**: 70:1 MUX가 완전히 사라지고 32비트 Shift Register 하나와 아주 작은 주변 제어 로직만 필요하므로, 조합회로 면적이 대폭 축소됩니다. (약 5~6만 Area 절감 예상)

### 💡 제안 3: Wishbone Cyc 신호 게이팅을 통한 인터록 시점 지연 (32비트 명령어 버퍼 완전 제거)
* **AS-IS**: 외부 메모리 페치는 즉시 개시하고, 들어온 명령어를 인터록 해제 시점까지 32비트 버퍼 `ibus_pending_rdt`에 대기시킴.
* **TO-BE**: GPR 쓰기 드레인 중일 때(`rf_write_drain_busy`가 활성화된 상태)는 Wishbone cycle의 시작 자체를 지연시킵니다.
  * 즉, 메모리 브릿지에 전달하는 ibus cycle 신호를 게이팅합니다: `i_ibus_cyc = wb_ibus_cyc && !rf_write_drain_busy;`
  * 인터록이 걸려 있는 동안에는 외부 메모리 접근 자체가 일어나지 않으므로, 다음 명령어가 미리 로드되어 대기할 필요가 없습니다.
  * 인터록이 해제되는 순간 메모리 접근이 시작되어 코어로 바로 전달됩니다.
* **효과**: 32비트 명령어 대기 버퍼 `ibus_pending_rdt`와 관련 레지스터들이 완전히 삭제됩니다. 이로 인해 무거운 `DFFSR` 플롭이 35개 이상 제거되어 `my_mcu_top` 자체 면적이 대폭 감소합니다. (약 5만 Area 절감 예상)

## 4. 기대 효과 및 결론

제안된 3가지 최적화 방안을 모두 적용할 경우 예상되는 면적 감소량은 다음과 같습니다.

| 최적화 대상 | 제안 내용 | 예상 면적 감소량 (Area 단위) |
| :--- | :--- | :---: |
| `serv_rf_ram_if` | Shift Register 구조 전환 (동적 MUX 제거) | ~90,000 |
| `offchip_mem_bridge` | 32비트 Shift Register 단계적 재사용 (70:1 MUX 제거) | ~55,000 |
| `my_mcu_top` | Wishbone Cyc 게이팅 인터록 (32비트 명령어 버퍼 제거) | ~50,000 |
| **합계** | **총 3가지 구조 최적화 동시 적용** | **~195,000 (약 22% 감소)** |

이 최적화가 적용되면 Yosys 합성 면적이 약 **89만에서 69만 수준**으로 급감하며, 이는 qflow 물리 구현 배치 시 실제 다이 면적(Die Area)을 **1.6mm² 이하**로 낮추고 배치 및 라우팅 난이도를 대폭 완화하는 데 결정적인 기여를 할 것입니다.
