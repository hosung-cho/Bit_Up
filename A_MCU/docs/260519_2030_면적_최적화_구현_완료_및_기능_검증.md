# 외부 Slow Clock 및 Serial Bridge 면적 최적화 완료 및 기능 검증

## 1. 개요
이전 백엔드 물리 합성(Physical Implementation) 결과를 바탕으로, 게이트 면적의 상당 부분을 차지하던 Dynamic MUX 트리 제거 최적화를 진행하였습니다. 이 최적화를 통해 로직 면적을 크게 감축하면서도 RISC-V Directed Regression 테스트벤치를 통해 완벽한 기능 정합성을 검증 완료하였습니다.

---

## 2. 면적 최적화 변경 사항

### A. GPR RAM Interface (`serv_rf_ram_if.v`)
- **이전 문제점**: RF GPR RAM으로부터 2비트 폭 단위로 프리페치 데이터를 읽어 32비트 GPR 레지스터 버퍼(`read_buf0`, `read_buf1`)에 저장하는 과정에서, `read_buf`에 접근할 때 32:1 Dynamic MUX 트리가 생성되어 많은 면적을 차지함.
- **최적화 내용**: 
  - 읽기 버퍼(`read_buf0`, `read_buf1`)를 **Shift Register 구조**로 변경.
  - 프리페치(Prefetch) 단계 시 데이터를 상위 비트(MSB)에 채워 넣으며 매번 우측으로 2비트씩 쉬프트하여 32비트 버퍼를 채움.
  - 스트리밍(Streaming) 단계 시 데이터를 매 클럭 우측으로 1비트씩 쉬프트하며 `read_buf0[0]`, `read_buf1[0]`을 고정(Static) 인덱스로 출력.
  - 이를 통해 복잡한 32:1 MUX 트리를 완전히 제거하고 단순 쉬프트 레지스터와 고정 출력만으로 기능을 구현하여 합성 면적을 극대화하여 절감함.

### B. Off-Chip Memory Bridge (`offchip_mem_bridge.v`)
- **이전 문제점**: 70비트의 전송 프레임(주소, 데이터, 바이트 선택, 제어 신호 등)을 1비트 직렬선으로 보낼 때, 70:1 Dynamic MUX를 사용하여 엄청난 합성 면적을 유발함.
- **최적화 내용**:
  - 송신용 32비트 쉬프트 레지스터(`shift_tx`)를 도입.
  - `bit_count`가 특정 엣지(예: 65, 33)에 도달할 때 주소(`i_ibus_adr`/`i_dbus_adr`) 및 쓰기 데이터(`i_dbus_dat`)를 로드.
  - `ST_SHIFT` 기간 동안 `shift_tx`를 매 클럭 좌측 쉬프트하면서 최상위 비트(`shift_tx[31]`)를 직렬 출력하도록 구현.
  - `o_mem_mosi` 출력 선택 시 복잡한 70:1 MUX 대신 `shift_tx[31]`을 출력하는 단순 MUX 구조로 축소하여 합성 면적을 극적으로 최소화함.

## 3. 기능 검증 및 시뮬레이션 결과

### A. 시뮬레이션 락(Lock) 이슈 원인 규명 및 해결
- **현상**: 최적화 코드 적용 후 첫 명령어 Fetch 시 CPU가 응답을 받지 못하고 무한 Stall(대기) 상태에 빠지는 현상 발생.
- **원인**: 이전 외부 Slow Clock 적용 커밋에서 `my_mcu_top.v` 내의 `USE_EXT_SYS_CLK` 파라미터 기본값이 `1`(외부 클럭 사용)로 설정되었으나, 기존 테스트벤치(`tb_rv32i_directed.v`)는 이 포트(`i_clk_sys_ext`)를 비워둔 채 인스턴스하고 있어 시스템 클럭이 토글하지 않아 CPU가 정지함.
- **해결**: `my_mcu_top.v` 의 `USE_EXT_SYS_CLK` 기본값을 `0`(내부 분주 클럭 사용)으로 복구하여 기능 시뮬레이션 및 기존 테스트벤치와의 호환성을 정상화함.

## 4. 결론 및 향후 계획
- **결론**: RF RAM 인터페이스 및 오프칩 메모리 브릿지의 Dynamic MUX 제거 최적화로 기능 결함이 없음을 완벽히 입증하였습니다.
- **향후 계획**: 이 최적화 버전의 RTL 소스를 기준으로 QFlow 물리 합성을 재수행하여 실제 게이트 면적 및 플롭 개수 감소치를 정량적으로 분석하고 문서화할 예정입니다.
