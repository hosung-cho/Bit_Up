# RV32I ISA Smoke 확장 및 현재 미해결 정리

작성 시각: 2026-05-16 17:43 KST

## 질문에 대한 답

현재 상태에서 "RISC-V 기본 ISA가 전부 실제로 돈다"고 말할 수는 없다.

기존 top-level smoke는 다음 범위를 확인했다.

- `addi`
- `lw/lb/lbu/lh/lhu`
- `sw/sb/sh`
- `beq/bne`
- `jal/jalr`
- CSR shadow `mscratch/mtvec/mepc/mtval`

하지만 RV32I 전체 관점에서는 아직 부족했다. 특히 `lui/auipc`, R-type 산술/논리/시프트, I-type 비교/논리/시프트, 전체 branch 6종을 한 번에 묶은 테스트가 없었다.

## 이번에 추가한 테스트

파일: `testbench/4_testbench_my_mcu_top/tb_my_mcu_top.v`

기존 smoke 프로그램 뒤에서 `jal x0, 0x200`으로 별도 ISA smoke 영역에 진입하도록 했다.

추가한 명령 범위:

- U-type: `lui`, `auipc`
- R-type: `add`, `sub`, `sll`, `slt`, `sltu`, `xor`, `srl`, `sra`, `or`, `and`
- I-type ALU: `slti`, `sltiu`, `xori`, `ori`, `andi`, `slli`, `srli`, `srai`
- Branch: `beq`, `bne`, `blt`, `bge`, `bltu`, `bgeu`

각 결과는 `0x180..0x1e4` 결과 메모리 영역에 `sw`로 남기고 테스트벤치가 기대값과 비교한다.

## 현재 시뮬레이션 결과

실행:

```sh
cd testbench/4_testbench_my_mcu_top/sim
./run.sh
```

기존 smoke는 계속 통과한다.

- load/store/GPIO smoke 통과
- branch/JAL/JALR 기존 smoke 통과
- CSR shadow `rf32..rf35` 통과

하지만 새 RV32I 확장 smoke는 실패한다.

대표 실패:

- `rf5`
  - observed `14000000`
  - expected `00000005`
- `rf8`
  - observed `e3ffffff`
  - expected `fffffff8`
- `lui -> store`
  - observed `2468a000`
  - expected `12345000`
- R-type/I-type 결과 저장 영역 다수 실패

중간에 시도한 수정:

- `rd_mem_en`을 `serv_top` 밖으로 노출
- `load`를 제외한 RD writeback에 next phase를 적용하는 실험

결과:

- `lui -> store` 일부는 개선됐지만 `jal/jalr` link 값이 깨졌다.
- 따라서 그 방식은 정식 해법이 아니어서 `i_wdata0_next` 연결은 기존 `rd_alu_en | rd_csr_en`으로 되돌렸다.
- `o_rd_mem_en` 노출 wiring은 남아 있지만 현재 phase 선택에는 사용하지 않는다.

## 현재 판단

문제는 코어의 RV32I decode 자체라기보다는, 오프칩 RF adapter에서 명령 종류별 RD writeback phase가 아직 충분히 일반화되지 않은 쪽에 가깝다.

지금까지는 `addi`, load, CSR, jump link 같은 일부 대표 경로만 맞춰 왔다. RV32I 전체 smoke를 넣자 다음 계열의 phase 차이가 추가로 드러났다.

- U-type writeback: `lui/auipc`
- R-type ALU writeback
- 일부 I-type ALU writeback
- 그 결과를 다시 store source로 읽는 경로

## 다음 작업

1. `serv_decode`의 writeback source 분류를 더 세밀하게 밖으로 노출한다.
   - `rd_alu_en`
   - `rd_csr_en`
   - `rd_mem_en`
   - `utype`
   - `jal_or_jalr`
   - 필요하면 `two_stage_op`/shift 계열
2. RF adapter의 `i_wdata0_next`를 단일 boolean이 아니라 source별 phase 선택으로 정리한다.
3. 작은 단위부터 다시 닫는다.
   - `addi x5, x0, 5`
   - `lui x7, ...`
   - `add x7, x5, x6`
   - `sw x7, ...`
4. 그 후 현재 추가한 RV32I smoke 전체를 다시 통과시킨다.

## 결론

현재 답은 명확하다.

- 기존 smoke 기준의 MCU 기능은 꽤 많이 통과했다.
- 하지만 RV32I 기본 ISA 전체가 오프칩 RF/메모리 구조에서 전부 돈다는 검증은 아직 통과하지 않았다.
- 이번 작업으로 그 사실을 드러내는 top-level ISA smoke가 추가됐고, 다음 핵심 과제는 writeback phase 분류를 명령 source별로 정식화하는 것이다.
