# 4핀 시리얼 RF 브리지 진행 상황

이 문서는 레지스터 파일(Register File, RF)을 칩 내부 RAM에서 제거하고, 외부 장치(Pico 등)에 위임하기 위한 현재 구현 상태를 정리한다.

검증 실행 기준은 `testbench/sim/run.bat`이다. Linux/Icarus 등 다른 방법으로 돌린 결과는 빠른 사전 점검으로만 취급하고, 공식 확인은 Vivado Tcl 흐름인 `run.bat`으로 한다.

현재 방향은 `serv_rf_top`을 사용하지 않고, 최상위 `my_mcu_top.v`에서 다음 구조로 직접 연결하는 것이다.

```text
my_mcu_top
  ├─ serv_top
  │   └─ SERV RF 포트
  ├─ serv_rf_ram_if
  │   └─ SERV RF 포트를 RAM 형태의 read/write access로 변환
  └─ serv_rf_ram
      └─ 내부 RAM 대신 4핀 시리얼 RF 브리지로 동작
```

즉, `serv_rf_ram.v`라는 파일명은 유지하지만 현재 의미는 "RAM"이 아니라 "Serial External RF Bridge"이다.

`serv_rf_top.v`는 현재 통합 방향에서 사용하지 않는다. `my_mcu_top.v`가 SoC 최상위이며, 여기서 `serv_top`, `serv_rf_ram_if`, 수정된 `serv_rf_ram`을 직접 묶는다.

## 현재 RTL 상태

### `my_mcu_top.v`

`my_mcu_top.v`에는 외부 RF 연결용 4핀 포트가 추가되어 있다.

```verilog
output wire o_rf_sync,
output wire o_rf_sck,
output wire o_rf_mosi,
input  wire i_rf_miso,
```

내부 연결은 다음 순서다.

```text
serv_top
  -> rf_wreq/rf_rreq, wreg/rreg, wdata/rdata
serv_rf_ram_if
  -> waddr/wdata/wen, raddr/ren/rdata
serv_rf_ram
  -> o_rf_sync/o_rf_sck/o_rf_mosi/i_rf_miso
```

클럭 구조는 현재 다음과 같다.

- `i_clk_16m`: 외부에서 들어오는 빠른 클럭. RF 시리얼 전송용 `i_clk_fast`로 사용한다.
- `clk_sys`: `i_clk_16m`을 분주한 SERV 시스템 클럭. 현재 의도는 약 1MHz 시스템 클럭이다.
- `rst`: `i_rst_n`을 반전한 active-high 리셋.

### `serv_rf_ram_if.v`

이 모듈은 원래 SERV의 SRAM 기반 RF 인터페이스다. 현재도 그대로 사용한다.

역할은 다음과 같다.

- SERV가 bit-serial로 내보내는 RF 접근을 `width=2` 단위 접근으로 묶는다.
- `o_waddr`, `o_wdata`, `o_wen`, `o_raddr`, `o_ren`을 만든다.
- `i_rdata`를 다시 SERV의 `i_rdata0`, `i_rdata1`로 되돌린다.

현재 설정은 다음 기준이다.

```verilog
localparam WITH_CSR  = 1;
localparam W         = 1;
localparam RF_WIDTH  = W * 2; // 2-bit access
localparam CSR_REGS  = WITH_CSR * 4;
localparam RF_DEPTH  = (32*(32+CSR_REGS))/RF_WIDTH;
```

### `serv_rf_ram.v`

현재 `serv_rf_ram.v`는 내부 레지스터 파일 RAM을 보유하지 않는다. 대신 `i_wen` 또는 `i_ren`이 발생하면 외부로 13비트 프레임을 전송한다.

현재 프레임 포맷은 다음과 같다.

```text
[12]    WEN
[11]    REN
[10:6]  REG index
[5:2]   BIT index
[1:0]   WDATA
```

전송은 MSB first다.

외부 핀 의미는 다음과 같다.

| 핀 | 방향 | 의미 |
| --- | --- | --- |
| `o_ext_rf_sync` | chip -> external | 프레임 구간 표시. High 동안 13비트 전송 |
| `o_ext_rf_sck` | chip -> external | 외부 장치가 샘플링할 시리얼 클럭 |
| `o_ext_rf_mosi` | chip -> external | 명령, 주소, write data |
| `i_ext_rf_miso` | external -> chip | read data |

현재 구현상 중요한 보정 사항은 다음과 같다.

- `i_clk_fast` 도메인에서 `i_clk` 상승 에지를 2단 동기화로 감지한다.
- `o_ext_rf_sck`는 `o_ext_rf_sync`가 High일 때 `~i_clk_fast`로 출력한다.
- 읽기 데이터는 마지막 2비트 구간에서 `i_ext_rf_miso`로 수신한다.
- x0 보호를 위해 read target이 x0이면 `o_rdata`는 0을 반환한다.

## 문서와 실제 코드 차이

이 문서의 초기 버전에는 단순한 1단 `clk_sys_d` 에지 감지와 `i_clk_fast` 그대로의 SCK 출력이 적혀 있었다.

현재 RTL은 그 뒤 수정되어 다음 차이가 있다.

- 시스템 클럭 에지 감지는 2단 동기화 방식이다.
- SCK는 `~i_clk_fast` 위상으로 출력한다.
- 테스트벤치도 이 위상 기준으로 Mock Pico가 동작하도록 맞춰져 있다.

따라서 아래 원칙을 둔다.

- 구현의 기준은 실제 RTL 파일 `src/rtl/serv/serv_rf_ram.v`이다.
- 이 문서는 구조, 현재 상태, 검증 계획을 기록한다.
- 문서 안에 과거 예시 코드가 있더라도, 최종 구현 여부는 RTL 파일과 `run.bat` 시뮬레이션 결과로 판단한다.

## 테스트 실행 규칙

앞으로 테스트벤치는 `testbench/sim/run.bat`을 통해 수행한다.

Windows/Vivado 환경 기준 실행 위치:

```bat
cd MPW_workspace\Bit_Up\testbench\sim
run.bat
```

현재 `run_sim.tcl`은 다음 흐름으로 동작한다.

- `../../src/rtl/serv/*.v` 로 SERV 관련 RTL 추가
- `../../src/rtl/*.v` 로 top RTL 추가
- `../*.v` 로 testbench 추가
- top testbench는 현재 `tb_my_mcu_top`
- Vivado xsim runtime은 `all`

주의할 점:

- `run_sim.tcl`은 `../*.v`를 모두 읽지만, 실제 실행 top은 `tb_my_mcu_top` 하나다.
- 다른 테스트벤치를 실행하려면 `run_sim.tcl`의 `set_property top ...` 값을 바꿔서 `run.bat`으로 실행한다.
- 현재 `tb_external_rf.v`는 오래된 포트 기준이라 파일셋에 포함될 경우 Vivado 컴파일을 방해할 수 있다. 삭제하거나 4핀 시리얼 방식으로 고쳐야 한다.

## 현재까지 확인된 검증

임시로 Icarus Verilog로 확인한 결과는 다음과 같다. 단, 이 결과는 공식 검증이 아니라 현재 상태 파악용이다.

- `tb_serial_external_rf`: 단독 RF 브리지 write/read loopback 통과
  - x10 bit[1:0]에 `2'b11` write
  - 같은 위치 read
  - CPU 측 수신값 `2'b11` 확인
- `tb_my_mcu_top`: 통합 top에서 RF frame 발생 확인
  - NOP instruction bus stub 사용
  - RF frame count가 0보다 큰 것 확인

앞으로 이 항목들은 `run.bat` 기준으로 다시 확인하고 기록한다.

하지만 이것만으로는 "레지스터 파일 오프칩화가 확실히 검증됐다"고 보기는 어렵다. 현재 검증은 프레임 발생과 단일 2비트 access 위주다.

## 반드시 추가해야 할 검증

레지스터 파일을 오프칩으로 빼는 작업은 CPU 실행 정확도에 직접 영향을 주므로, 최소한 아래 항목은 확인해야 한다.

1. `serv_rf_ram` 단독 프로토콜 검증
   - write frame의 `[12:0]` 비트가 의도대로 나오는지 확인
   - read frame에서 외부 `MISO` 2비트가 정확히 `o_rdata`로 들어오는지 확인
   - x0 read는 항상 0인지 확인
   - x1, x2, x10, x31처럼 주소 경계가 다른 레지스터를 확인
   - bit index `0, 2, 4, ..., 30` 전체 또는 대표값 확인

2. `serv_rf_ram_if + serv_rf_ram` 결합 검증
   - SERV RF 접근 한 번이 필요한 수의 2비트 프레임으로 변환되는지 확인
   - `rdata0`, `rdata1` 타이밍이 SERV 기대와 맞는지 확인
   - 연속 write 후 read-back 확인

3. `my_mcu_top` 통합 검증
   - 단순 NOP frame 발생 확인을 넘어서 실제 명령 실행 기반 검증 필요
   - 예: `addi x10, x0, imm`, `addi x11, x10, imm` 같은 레지스터 의존 명령 실행
   - 외부 Mock Pico RF memory에 기대 레지스터 값이 쌓이는지 확인

4. 타이밍 검증
   - 현재 전제는 fast clock 16MHz, system clock 약 1MHz다.
   - 13비트 프레임이 SERV가 값을 필요로 하는 window 안에 끝나는지 확인해야 한다.
   - `i_rf_ready`와 `o_rdata` 유효 시점이 실제 SERV state machine과 충돌하지 않는지 파형으로 확인해야 한다.

5. Vivado 기준 검증
   - 최종 확인은 반드시 `run.bat`으로 수행한다.
   - Icarus Verilog 결과는 빠른 사전 점검으로만 취급한다.

## 현재 남아 있는 문제

### `tb_external_rf.v`는 오래된 테스트벤치

`tb_external_rf.v`는 예전 병렬 외부 RF 포트 기준이다.

예전 포트 예:

```verilog
o_ext_rf_rreg
o_ext_rf_wreg
o_ext_rf_wdata
o_ext_rf_bit_idx
i_ext_rf_rdata
o_ext_rf_wen
o_ext_rf_ren
```

현재 `serv_rf_ram.v`에는 위 포트들이 없다. 따라서 이 테스트벤치는 삭제하거나 4핀 시리얼 프로토콜 기준으로 다시 작성해야 한다.

### `serv_rf_top.v`는 현재 방향에서 제외

현재 프로젝트 방향은 `serv_rf_top`을 쓰지 않는다.

이유:

- `my_mcu_top`에서 `serv_top + serv_rf_ram_if + serv_rf_ram`을 직접 통합하고 있다.
- `serv_rf_top`은 기존 내부 RF RAM 중심 wrapper 성격이 강하다.
- 현재 수정된 `serv_rf_ram`은 fast clock과 외부 4핀 포트가 필요한데, `serv_rf_top`은 그 구조를 반영하지 않는다.

따라서 `serv_rf_top`은 참조용으로만 두고, 합성/통합 대상에서는 제외하는 방향이 맞다.

현재 사용할 통합 경로:

```text
my_mcu_top.v
  serv_top
    RF request/data ports
  serv_rf_ram_if
    SERV RF protocol -> 2-bit RAM-like RF access
  serv_rf_ram
    2-bit RF access -> 4-pin serial external RF frame
```

사용하지 않을 경로:

```text
serv_rf_top.v
  기존 SERV RF wrapper path
```

### 명령어 메모리와 데이터 메모리는 아직 미연결

`my_mcu_top.v`에서 RF 오프칩화는 연결되어 있지만, instruction/data memory 쪽은 아직 TODO 상태다.

현재 상태:

- instruction bus는 실제 외부 메모리 브리지에 연결되어 있지 않다.
- testbench에서 NOP와 ACK를 강제로 넣어 RF frame 발생만 확인한다.
- data memory 영역 `sel_mem`은 `ack=0`으로 되어 있어 실제 load/store는 진행되지 않는다.

즉, "RF 오프칩 브리지"와 "외부 명령어/데이터 메모리"는 아직 별개의 작업이다.

## 앞으로 해야 할 일

1. 테스트 실행 경로 정리
   - 모든 공식 테스트는 `testbench/sim/run.bat` 기준으로 수행한다.
   - `run_sim.tcl`에서 필요한 top testbench를 명시적으로 바꿔가며 검증할 수 있게 정리한다.

2. 오래된 테스트벤치 정리
   - `tb_external_rf.v`를 삭제하거나 `tb_serial_external_rf.v`와 같은 4핀 시리얼 방식으로 교체한다.
   - `tb_serial_external_rf.v`는 단독 프로토콜 검증용으로 유지한다.
   - `tb_my_mcu_top.v`는 top 통합 검증용으로 확장한다.

3. RF 브리지 단독 검증 강화
   - 여러 레지스터 주소와 bit index에 대해 write/read를 반복한다.
   - x0 보호 동작을 확인한다.
   - 연속 transaction을 확인한다.
   - reset 직후 첫 transaction을 확인한다.

4. SERV 통합 검증 강화
   - NOP만 공급하는 테스트에서 벗어나 실제 레지스터 의존 명령을 실행한다.
   - 외부 Mock Pico RF memory 값을 기준으로 레지스터 write/read가 맞는지 확인한다.

5. 타이밍 파형 확인
   - `rf_wreq`, `rf_rreq`, `rf_ready`, `ren`, `wen`, `rdata`, `sync`, `sck`, `mosi`, `miso`를 파형으로 확인한다.
   - 13비트 프레임 완료 시점과 SERV가 read data를 소비하는 시점이 맞는지 확인한다.

6. `serv_rf_top` 사용 경로 제거 또는 격리
   - 현재 방향에서는 `serv_rf_top`을 쓰지 않는다고 명확히 둔다.
   - Vivado 파일셋에서 top이 잘못 잡히지 않도록 `my_mcu_top`/`tb_my_mcu_top` 중심으로 유지한다.

7. 명령어 메모리/데이터 메모리 오프칩화 설계
   - RF 브리지 검증 후, Wishbone instruction/data bus를 외부 메모리 브리지에 연결한다.
   - instruction fetch와 data load/store를 같은 외부 장치로 보낼지, 별도 프로토콜로 나눌지 결정한다.
   - 현재 `my_mcu_top.v`의 `sel_mem` TODO를 실제 bridge 응답으로 교체한다.

## 현재 결론

레지스터 파일 오프칩화는 "방향성과 1차 브리지 구현"까지는 진행됐다.

다만 현재 검증은 아직 충분하지 않다. 특히 SERV가 실제 명령을 실행하면서 외부 RF를 정상 레지스터 파일처럼 사용하는지 확인해야 한다.

다음 우선순위는 다음과 같다.

1. `run.bat` 기준으로 현재 `tb_my_mcu_top`이 안정적으로 도는지 확인한다.
2. `tb_external_rf.v`를 현재 4핀 구조에 맞게 정리하거나 파일셋에서 제외한다.
3. `tb_serial_external_rf.v`를 강화해서 RF 브리지 단독 검증을 촘촘히 만든다.
4. `tb_my_mcu_top.v`를 실제 명령 실행 검증으로 확장한다.
5. `serv_rf_top`이 아닌 `my_mcu_top` 경로만 합성/시뮬레이션 대상으로 유지한다.
6. 그 다음 instruction/data memory 오프칩 브리지 설계로 이동한다.
