# 오프칩 Register File 통합 검증 진행 정리

작성 시점: 2026-05-16 02:12 KST

이 문서는 레지스터 파일(Register File, RF)을 칩 내부에서 제거하고 외부 장치(Pico 등)에 위임하는 작업의 현재 진행 상황을 정리한다. 현재 검증 실행 기준은 `run.sh`이다.

## 현재 방향

현재 방향은 `serv_rf_top`을 사용하지 않는다.

최상위인 `my_mcu_top.v`에서 다음 구성을 직접 묶는다.

```text
my_mcu_top
  ├─ serv_top
  │   └─ SERV RF native port
  ├─ serv_rf_ram_if
  │   └─ SERV RF port를 width=2 RAM-style access로 변환
  └─ serv_rf_ram
      └─ 내부 RF RAM 대신 4핀 시리얼 외부 RF 브리지로 동작
```

`serv_rf_ram.v`라는 파일명은 유지하고 있지만, 현재 의미는 내부 RAM이 아니라 `Serial External RF Bridge`에 가깝다.

## 실행 규칙

앞으로 테스트벤치는 각 테스트벤치 폴더의 `sim/run.sh`로 실행한다.

예:

```bash
cd MPW_workspace/Bit_Up/testbench/4_testbench_my_mcu_top/sim
./run.sh
```

Vivado xsim은 사용자 홈의 Xilinx cache/state 디렉터리에 접근하므로, Codex 환경에서는 `./run.sh` 실행 시 sandbox 밖 실행 권한이 필요했다. 사용자의 일반 터미널에서는 그대로 실행 가능하다.

기존 문서 중 `run.bat`을 공식 기준으로 적어둔 내용은 현재 기준과 맞지 않는다. 현재 공식 기준은 `run.sh`이다.

## 현재 RTL 상태

### `my_mcu_top.v`

`my_mcu_top.v`는 SoC 최상위로 사용한다.

외부 RF용 4핀 포트:

```verilog
output wire o_rf_sync,
output wire o_rf_sck,
output wire o_rf_mosi,
input  wire i_rf_miso
```

내부 연결:

```text
serv_top
  -> rf_wreq/rf_rreq
  -> wreg0/wreg1, wen0/wen1, wdata0/wdata1
  -> rreg0/rreg1, rdata0/rdata1

serv_rf_ram_if
  -> waddr/wdata/wen
  -> raddr/ren/rdata

serv_rf_ram
  -> o_rf_sync/o_rf_sck/o_rf_mosi/i_rf_miso
```

클럭 입력은 특정 주파수 이름을 박아두지 않고 `i_clk_fast`로 둔다. 실제 주파수는 보드/패키지/외부 oscillator 결정에 따른다.

현재 `clk_sys`는 `i_clk_fast`를 parameter로 분주해서 만든다.

```verilog
module my_mcu_top #(
    parameter integer CLK_SYS_DIV = 32
) (
    input wire i_clk_fast,
    ...
);
```

분주 관계:

```text
clk_sys = i_clk_fast / CLK_SYS_DIV
```

예를 들어 `CLK_SYS_DIV=32`이면 `clk_sys`는 `i_clk_fast`의 1/32이다. `i_clk_fast`가 16MHz라면 `clk_sys`는 500kHz가 된다. `i_clk_fast`가 8MHz라면 `clk_sys`는 250kHz가 된다.

분주가 필요한 이유는 RF serial frame이 system clock 한 구간 안에서 끝나야 하기 때문이다. 현재 RF bridge는 1개의 2비트 RF chunk 접근을 13비트 serial frame으로 전송한다.

```text
1 RF chunk access = 13 serial bits
```

따라서 최소 조건은 다음처럼 볼 수 있다.

```text
CLK_SYS_DIV >= 13 + margin
```

`CLK_SYS_DIV=16`이면 13비트 전송이 겨우 들어가며 여유가 작다. `CLK_SYS_DIV=32`이면 bring-up 단계에서 더 안전한 여유를 갖는다. 단, `CLK_SYS_DIV`를 키우면 SERV 실행 속도는 그만큼 느려진다.

권장 흐름:

```text
초기 bring-up: CLK_SYS_DIV=32
동작 확인 후 속도 상향: CLK_SYS_DIV=16 검토
최종 고속화: RF bridge에 busy/done handshake를 추가해서 고정 분주비 의존성 제거
```

리셋 관련 수정도 들어갔다. 기존에는 `rst = ~i_rst_n`에 가까운 구조라서 `clk_sys`가 reset release 이후에야 움직이는 구조와 충돌할 수 있었다. 현재는 `clk_sys` 도메인에서 reset stretcher를 사용한다.

```verilog
reg [3:0] rst_shift;
always @(posedge clk_sys or negedge i_rst_n) begin
    if (!i_rst_n)
        rst_shift <= 4'hf;
    else
        rst_shift <= {rst_shift[2:0], 1'b0};
end
wire rst = |rst_shift;
```

이 수정 후 `my_mcu_top` 테스트에서 instruction fetch와 RF frame이 정상적으로 발생했다.

### `serv_rf_ram.v`

`serv_rf_ram.v`는 내부 RAM을 갖지 않고, `i_wen` 또는 `i_ren` 요청을 13비트 serial frame으로 외부에 내보낸다.

프레임 포맷:

```text
[12]    WEN
[11]    REN
[10:6]  REG index
[5:2]   CHUNK index
[1:0]   WDATA
```

중요한 점은 `[5:2]`가 직접 bit offset이 아니라 2비트 chunk index라는 점이다. 외부 RF memory는 다음처럼 해석한다.

```verilog
bit_offset = chunk_index * 2;
```

현재 동작:

- `sync` high 동안 13비트 전송
- MSB first
- `sck`는 `sync` high 동안 `~i_clk_fast`
- `mosi`로 frame 전송
- read frame의 마지막 2비트 구간에서 `miso`를 받아 `o_rdata`로 전달
- x0 read는 항상 0으로 보호

초기 구현은 `clk_sys` rising edge 감지에 기대어 frame을 시작했지만, 이후 request-change 기반 launch 방식으로 수정했다. 이 덕분에 같은 `clk_sys` 구간 안에서 주소/데이터가 안정된 뒤 frame이 시작된다.

### `serv_rf_ram_if.v`

이 모듈이 이번 디버깅의 핵심이었다.

처음에는 외부 serial RF를 일반 SRAM처럼 즉시 read 가능한 것으로 취급하려 했지만, 실제로는 serial read latency 때문에 `i_rdata`를 즉시 줄 수 없다. 따라서 read side는 SERV를 `i_rf_ready`로 기다리게 한 뒤, 필요한 두 source register를 외부 RF에서 미리 읽어오는 방식으로 바꿨다.

현재 구조:

- write side
  - 원래 SERV RAM adapter의 write chunk 생성 타이밍을 유지한다.
  - `i_wreq`, `i_wen0`, `i_wen1`, `i_wdata0`, `i_wdata1`의 SERV 타이밍을 그대로 사용해 `width=2` write access를 만든다.
  - 별도 capture/flush 방식은 제거했다. 이 방식은 SERV의 실제 write timing과 어긋나 x값 또는 잘못된 write를 만들 수 있었다.

- read side
  - `i_rreq`가 들어오면 `i_rreg0`, `i_rreg1`을 latch한다.
  - 외부 RF에 32개의 2비트 read request를 낸다.
    - reg0 chunk 0
    - reg1 chunk 0
    - reg0 chunk 1
    - reg1 chunk 1
    - ...
    - reg0 chunk 15
    - reg1 chunk 15
  - 읽은 값을 `read_buf0`, `read_buf1`에 저장한다.
  - prefetch가 끝난 뒤 `ready_pulse`를 내보낸다.
  - `ready_pulse` 이후 1 sys-clock 늦춰 `o_rdata0/o_rdata1` stream을 시작한다. 이 지연이 없으면 SERV counter보다 read data가 한 비트 빠르게 흘러 x10=3을 x10=1처럼 읽는 문제가 있었다.
  - x0 read는 어댑터에서도 0으로 강제한다.
  - write 직후 read prefetch가 너무 빨리 시작하지 않도록 `write_wait` window를 둔다.

결론적으로 `serv_rf_ram_if`는 이제 단순 SRAM adapter가 아니라 external serial RF latency를 SERV의 `i_rf_ready` 프로토콜로 흡수하는 adapter가 되었다.

### 2. `3_testbench_rf_if_serial`

목적: `serv_rf_ram_if + serv_rf_ram` 결합 검증.

검증 내용:

- SERV-like write request가 2비트 chunk write frame으로 변환되는지 확인
- x5 = `32'hA5A5_3C3C` write
- x6 = `32'h5A5A_C3C3` write
- 외부 mock RF memory에 정확히 저장되는지 확인
- read prefetch 후 `o_rdata0/o_rdata1` stream 안에 기대 word가 존재하는지 확인

중간에 발견한 문제:

- testbench가 `i_wen0/i_wen1`을 너무 일찍 올려 write capture가 앞당겨지는 race가 있었다.
- `serv_rf_ram_if`에서 직접 capture/flush를 만들면 SERV의 실제 write timing과 어긋났다.
- write side는 원래 adapter timing을 유지하는 것이 맞았다.
- read stream은 `ready` 직후 즉시 시작하면 SERV counter보다 한 비트 빠르게 보였다.

최종 실행:

```bash

./run.sh
```

결과:

```text

[TB PASS] rf_if + serial bridge integration passed. frames=64 writes=32 reads=32
```

### 3. `4_testbench_my_mcu_top`

목적: `my_mcu_top`에서 실제 SERV 명령 실행이 external RF 경로를 통해 맞는지 확인.

테스트 프로그램:

```text
addi x10, x0, 3
addi x11, x10, 4
add  x12, x10, x11
jal  x0, 0
```

기대값:

```text
x10 = 3
x11 = 7
x12 = 10
```

중간에 발견한 문제:

- reset release 후 `clk_sys`에서 SERV가 reset을 제대로 보지 못할 수 있어 reset stretcher를 추가했다.
- instruction bus를 combinational force로 바로 응답시키면 주소/명령 안정 전 샘플링으로 X가 섞일 수 있었다.
- testbench instruction bus를 `clk_sys` 기준 1-cycle response로 바꿨다.
- write frame은 나오지만 다음 instruction read가 이전 write 반영 전에 시작되어 부분 값만 읽는 문제가 있었다.
- `serv_rf_ram_if`에 write settle window를 추가했다.
- read data stream이 SERV counter보다 빠르게 시작되어 x10=3을 x10=1처럼 읽는 문제가 있었다.
- `ready_pulse` 이후 read stream 시작을 1 sys-clock 늦춰 정렬했다.

## 아직 남은 주의점

현재 통과한 것은 behavioral simulation 기준이다. 다음 항목은 아직 별도 확인이 필요하다.

1. 실제 Pico firmware와의 cycle-level protocol 확인
   - 현재 mock Pico는 testbench 로직이다.
   - 실제 Pico가 `sync/sck/mosi/miso` 타이밍을 같은 방식으로 처리해야 한다.

2. timing 여유 확인
   - 특정 fast clock 주파수는 아직 정하지 않았다.
   - 현재 RTL은 `CLK_SYS_DIV` parameter로 `clk_sys = i_clk_fast / CLK_SYS_DIV`를 만든다.
   - 기본값은 bring-up 여유를 위해 `CLK_SYS_DIV=32`이다.
   - 13비트 frame은 fast clock 기준으로 한 sys-clock 안에 끝난다는 전제에 기대고 있으므로 `CLK_SYS_DIV >= 13 + margin`을 만족해야 한다.

3. 더 많은 instruction sequence
   - 현재 top 검증은 `addi/add/jal` 중심이다.
   - load/store, branch, CSR, trap/mret이 필요하면 별도 테스트가 필요하다.

4. CSR register 영역
   - `WITH_CSR=1`이라 RF address space에 CSR 4개가 붙는다.
   - 현재 검증은 GPR 중심이다.

5. off-chip instruction memory/data memory
   - 이번 작업은 register file off-chip화 검증이다.
   - instruction memory와 data memory off-chip화는 다음 단계에서 별도 bridge/protocol과 stall 정책이 필요하다.

6. timescale warning
   - Vivado에서 여러 RTL 파일에 `timescale`이 없다는 warning이 나온다.
   - 기능 실패는 아니지만, 필요하면 RTL 파일들에 일관된 `` `timescale 1ns/1ps ``를 추가할 수 있다.

## 현재 결론

`serv_rf_top`을 쓰지 않고 `my_mcu_top`에서 `serv_top + serv_rf_ram_if + serv_rf_ram`으로 직접 가는 방향은 현재 behavioral simulation에서 동작한다.

특히 중요한 결론은 다음과 같다.

- write side는 원래 SERV RF RAM adapter의 timing을 유지해야 한다.
- read side는 external serial latency 때문에 prefetch와 `i_rf_ready` stall이 필요하다.
- `o_ready`와 `o_rdata` stream의 상대 타이밍은 SERV counter와 맞춰야 하며, 현재는 `ready_pulse` 이후 1 sys-clock 지연이 맞다.
- write 직후 read는 외부 반영이 끝난 뒤 시작해야 하므로 write settle window가 필요하다.

다음 큰 작업은 instruction memory와 data memory를 off-chip으로 빼는 구조를 별도로 설계하고, RF bridge와 함께 top-level pin/protocol을 정리하는 것이다.
